Architecture: Upcoming Elections Feature

How the pieces fit together — pick the approach that feels right

A

Separate Elections Service + Shared Research Pipeline

New backend: services/elections.py calls Google Civic API voterinfo endpoint. Returns elections, contests, and candidates as new models. New router routers/elections.py with POST /api/elections.

Research reuse: Candidate cards use the existing POST /api/research endpoint as-is — a candidate is just a Representative model with a different source. New election research endpoint for per-election context (lighter pipeline, 2-3 sections).

Frontend: React Router added. Two routes after search: /reps and /elections. Tab bar switches between them. Elections page groups contests by election, renders candidate cards (same RepCard component).

Pros

  • Clean separation — elections logic is isolated
  • RepCard reuse means minimal new UI
  • Election research is a small, focused pipeline

Cons

  • Two research endpoints (rep + election) to maintain
  • Google Civic API can be sparse outside election season
B

Unified Lookup + Research with Election Context Layer

Backend: Same new services/elections.py, but elections data is fetched alongside reps in a single expanded lookup. POST /api/representatives returns both current reps AND upcoming election candidates in one response.

Research: Single research endpoint handles both reps and candidates. Election-level context is a new "election research" task type in the existing pipeline — same store, same polling pattern.

Frontend: Same router setup, but data comes from one API call. Tabs filter the unified response.

Pros

  • Single API call from frontend
  • Unified data model

Cons

  • Bloats the rep lookup endpoint with election concerns
  • Slower initial load (waiting on Civic API too)
  • Harder to evolve independently
C

Lazy-Load Elections (Recommended)

Backend: New services/elections.py + routers/elections.py with POST /api/elections. Completely independent from rep lookup. New election research pipeline (2-3 sections) with its own endpoint POST /api/election-research.

Research reuse: Candidate research uses existing POST /api/research unchanged. Election-level research is new but follows the same patterns (background task, polling, incremental sections).

Frontend: React Router with /reps and /elections. Rep data loads immediately on search. Election data loads only when user clicks the Elections tab — lazy fetch. This keeps the happy path fast.

Pros

  • Zero impact on existing rep lookup speed
  • Clean separation — elections is a self-contained feature
  • Election tab can show a loading state independently
  • Easiest to build incrementally and test in isolation

Cons

  • Two API calls from frontend (one per tab)
  • Slight delay when switching to Elections tab first time
